01
How to Use This SOP
Read before touching any org structure

The supervisory org hierarchy is the structural backbone of Workday. Every security constraint, every business process routing, every report filter, and every role assignment is anchored to this hierarchy. A poorly designed or incorrectly maintained org structure cascades into failures across security, payroll, reporting, and user experience simultaneously.

🔴 Critical
Moving orgs, reassigning managers, or restructuring the hierarchy affects: (1) role-based security group membership — immediately, (2) business process routing — immediately, (3) reporting rollups — immediately, (4) compensation hierarchy for review cycles. Always model the impact across all four dimensions before executing any org change in production.
✅ Tip
Maintain a visual org chart (Visio, Miro, or org chart export from Workday) alongside this SOP. The visual is the first reference point during restructure planning — having it current and accessible reduces planning errors significantly.
02
Org Architecture Primer
Org types, hierarchy rules, and design principles
Supervisory Org

The primary org type. Reflects the management hierarchy — workers belong to supervisory orgs. Security constraints, manager relationships, and BP routing all derive from this structure.

Company / Legal Entity

A legal-entity org type used for payroll, legal reporting, and financial consolidation. Workers have a primary company assignment. Does not drive security or BP routing.

Cost Center

Financial org type. Workers (or positions) are assigned a cost center for financial reporting and budget allocation. Cost center hierarchy is separate from supervisory org hierarchy.

Region

Geographic org type. Used for grouping workers by geography for reporting, headcount analysis, and regional HR operations. Drives constrained security for regional HR Leads.

Custom Org Types

Business-defined org types (Business Unit, Division, Function, Practice). Used for reporting segmentation and cross-matrix structures. Workers can belong to multiple custom org types simultaneously.

Matrix Orgs

Workday supports matrix management via additional org assignments (e.g., a worker in a Supervisory Org under their direct manager, and a Project Org under a dotted-line lead). Matrix orgs do not drive primary security.

💡 Rationale
The supervisory org hierarchy should reflect the management reporting line — not the business or functional structure. Finance, Division, and Business Unit structures should use separate Cost Center or Custom Org hierarchies. Conflating management and financial structure in the supervisory org is the most common org design error in Workday implementations.

04
Supervisory Org Creation & Configuration
Building new orgs and configuring their properties
1
Create the Supervisory Organisation
Search Create Supervisory Organization. Enter the org name per your naming convention. Set the Availability Date — the earliest date the org can have workers assigned. Set the Staffing Model: Position Management (positions exist independently of workers) or Job Management (positions created with workers). Confirm with your HRIS lead which model your tenant uses — do not mix models.
🔴 Critical
Staffing Model selection at org creation is permanent — it cannot be changed after workers or positions are assigned to the org. Defaulting to the wrong model requires a full org deletion and rebuild. Confirm the tenant's staffing model before creating any org.
2
Assign the Manager
Within the org setup, assign the Manager — the worker who leads this team. The manager assignment drives: (1) self-service access to team members, (2) approval routing in BPs, (3) Manager security group membership, (4) compensation review org grouping. Confirm the manager's worker record exists and is active before assigning.
✅ Tip
If the new manager is being hired simultaneously with the org creation (common in M&A), create the manager's worker record first — even as a placeholder — before creating the org. Org creation requires an existing Workday worker as manager. You cannot assign a manager who does not yet exist in the system.
3
Set the Parent Org
Assign the Superior Organization — the org one level up in the hierarchy. This determines the inheritance chain for security, BP routing, and reporting rollups. Navigate the org tree to select the correct parent. Confirm the parent org exists and is correctly positioned in the hierarchy before saving.
⚠️ Warning
Setting the wrong parent org is one of the most impactful single-field errors in Workday. It immediately misroutes security access, approvals, and reporting. Double-check the parent org by viewing the org hierarchy before and after setting the parent.
4
Configure Org Assignments (Cost Center, Region, Custom Orgs)
After creating the supervisory org, navigate to Edit Organization Assignments. Assign: Cost Center, Company / Legal Entity, Region, and any Custom Org types (Business Unit, Division, Function). These assignments populate automatically on worker records when they are placed in this org — simplifying data entry and ensuring consistency.
📋 Note
Cost Center and Company assignment on the supervisory org sets the default for workers in that org. Individual workers can be assigned a different Cost Center at the position level if needed. Confirm with Finance whether Cost Center assignment should be at the org or position level for your tenant's financial reporting model.
5
Assign Locations
Navigate to the org → Locations. Add the physical location(s) where this org's workers are based. Location assignment at the org level sets the default for new positions. For orgs spanning multiple locations (common in global teams), add all relevant locations and confirm with HR Partners that workers are being assigned the correct individual location during Hire transactions.
05
Role Assignments on Orgs
Assigning HR Partners, Recruiters, and other roles to supervisory orgs

Role assignments connect workers to role-based security groups (see Vol. 02) by assigning them a named role on a supervisory org. A worker assigned the HR Partner role on an org gains the security access and BP permissions of the HR Partner security group — constrained to workers within that org hierarchy.

1
Assign a Role to the Org
Navigate to the supervisory org → Assign Roles. Select the role (e.g., HR Partner, Recruiter, Benefits Partner). Search for the worker to assign. Set the scope — does this role assignment cover just this org, or this org and all subordinate orgs? For HR Partners, typically select "include subordinate organisations" to cover the full reporting hierarchy.
✅ Tip
Assign HR Partner roles at the highest org level the HR Partner supports — the subordinate org inclusion means they automatically get access to all child orgs without individual assignments. Avoid assigning the same HR Partner to 50 individual orgs when one top-level assignment with subordinate inclusion achieves the same result with far less maintenance.
2
Validate Role Assignment Takes Effect
After assigning the role, confirm the assignment is active: navigate to the assigned worker's profile → Security Roles tab. Confirm the role appears with the correct org scope. Then use View As (Vol. 02, Section 06) to validate the worker now has the expected security access for workers within that org hierarchy.
⚠️ Warning
Role assignments on orgs are immediate — the worker gains security access as soon as the assignment is saved. If you are pre-configuring for a future restructure, time role assignments carefully. A worker should not gain access to an org's workers before the restructure is communicated and effective.
3
Remove Role Assignments When No Longer Needed
When an HR Partner leaves, transfers, or the org restructures, remove role assignments immediately. Navigate to the org → Assign Roles → remove the worker from the role. This immediately revokes the security access associated with the role — no additional security steps required.
🔴 Critical
Stale role assignments — HR Partners or Recruiters who have left or transferred but still hold roles on orgs — are a top-5 Workday security audit finding. Include role assignment review in your offboarding checklist (Hire to Retire BP To-Do) for all workers in HR and Talent Acquisition roles.
06
Additional Org Types
Cost Centers, Regions, Company, and Custom Org configuration
Cost Center Configuration
1
Create Cost Center
Search Create Cost Center. Enter the cost center ID (must match your Finance system's GL code exactly — any mismatch breaks financial reporting integrations). Enter the name and assign the Cost Center Hierarchy parent. Assign a Cost Center Manager — typically the budget owner for that cost center.
⚠️ Warning
Cost Center IDs feed financial system integrations. A typo in the Cost Center ID on creation can result in payroll charges being attributed to the wrong GL code in your ERP. Validate the ID against your Finance system's chart of accounts before saving.
2
Assign Cost Center to Supervisory Org
Navigate to the supervisory org → Edit Organization Assignments → Cost Center field. Select the new cost center. This populates as the default for all workers placed in this org going forward. Existing workers retain their current cost center until individually updated or mass updated via EIB (see Vol. 01).
Custom Org Types

Custom org types (Division, Business Unit, Function, Practice) are configured via Maintain Organization Types. Each type has its own hierarchy. Workers can be assigned to multiple custom org types simultaneously — creating the matrix view Finance and leadership often need for cross-functional reporting. Configure custom org types by:

  • Creating the org type definition (Maintain Organization Types)
  • Creating the org instances within each type (Create Organization → select type)
  • Assigning workers or positions to the custom org via Edit Org Assignments or EIB mass load
  • Building reports and dashboards using the custom org as a filter or grouping dimension (Vol. 05)

Restructures, Moves & M&A Scenarios
High-complexity org changes — planning and execution
🔴 Critical
Org restructures are the highest-risk org management scenario in Workday. Moving workers or orgs across hierarchy levels, legal entities, or cost centers triggers cascading impacts across payroll, security, BPF, integrations, and reporting simultaneously. Never execute a restructure without a written impact assessment reviewed by HRIS, Finance, Legal, and HR Operations.

Moving Workers Between Orgs
1
Use Transfer / Move Worker (Not Terminate + Rehire)
For workers moving between supervisory orgs within the same legal entity, use the Move Workers task or initiate a Transfer business process. Do not terminate and rehire — this breaks service continuity, resets tenure, and creates data integrity issues in benefits and compensation history. Transfer preserves the full worker record and employment history.
2
For Cross-Entity Moves — Use Global Transfer
Workers moving between legal entities require a Global Transfer (not Transfer). Global Transfer creates a new employment record in the target entity while maintaining the relationship to the prior employment. Configure: new position, new cost center, new compensation if applicable, and new legal entity. Confirm the tax and payroll implications with Finance before initiating.
⚠️ Warning
Global Transfer initiates a new Hire BP in the target entity in addition to the termination BP in the source entity. Both BPs must complete for the transfer to be clean. Monitor both in-flight simultaneously and confirm all approvals clear before the worker's transfer effective date.
3
Post-Move Validation
After executing moves: (1) confirm workers appear in the correct org in Workday, (2) validate manager self-service shows the moved workers under new manager, (3) confirm HR Partner role access is correct for the new org, (4) run a headcount report pre/post to validate counts balance, (5) confirm integrations have picked up the move in the next scheduled run.

09
Validation & Testing
Confirming org changes work as designed
  • New org appears in org hierarchy at correct level — confirmed via Workday org chart view
  • Manager assignment confirmed — manager's self-service shows workers in the org
  • HR Partner role assignment confirmed — HR Partner can view all workers in the org via View As
  • Cost Center and org assignments populate correctly on test Hire transaction
  • BP routing validated — approval routes to correct manager for test transaction in new org
  • Headcount report shows moved workers in correct org from effective date
  • Integration output reflects org changes on next scheduled run — no orphaned or misrouted records
  • For cross-entity moves: payroll confirms worker appears in correct entity's payroll run
